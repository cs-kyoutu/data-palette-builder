# ヒアリングガイド（技術確認のみ）

## 基本方針

Phase1の役割は**b→dashの技術的な確認**と**定義不足の指摘**。
要件（対象期間、件数、優先順位、除外条件等）は**アウトプット定義の「定義」列に記載済みの前提**。
記載がなければ**推論せず「情報不足」として回答する**。勝手に補完してはいけない。
技術確認（リレーション項目の番号、データ形式等）は**質問してOK**。

---

## 聞かなくていいこと（デフォルトルール）

以下はb→dashの標準運用として自明。質問不要で自動適用：

| 項目 | デフォルト |
|------|-----------|
| 顧客テーブルとの結合方法 | 「先に選択したデータに対して統合する」（LEFT JOIN）固定。顧客は全量残す |
| 名前の結合方法 | 姓 + スペース + 名の連結で固定 |
| 閲覧/カート投入の判定ロジック | 該当リレーション項目に値がある = 対象レコード（自明） |
| 不要カラムの削除 | 聞かない。中間カラムは最終的に整理 |
| 商品情報が見つからない場合 | 除外。検証観点に「商品マスタに存在しない商品IDの件数」を入れる |
| 対象期間 | アウトプット定義の「定義」列から読み取る。**書いてなければ情報不足として回答** |
| 上位N件 | アウトプットに「1〜5」等あれば自明（5件）。**なければ情報不足として回答** |
| 優先順位 | アウトプット定義に記載されていればそのまま採用。「最新」「最終」→降順、「初回」→昇順は自明 |
| 除外条件 | アウトプット定義に記載されていればそのまま採用 |
| 結合方法 | bizデータ同士は「共通のデータのみを統合する」、顧客テーブルは「先に選択したデータに対して統合する」 |
| 「最終」「初回」の1件特定 | 名寄せで対象キーの最新/最古を優先 |
| 複数明細がある場合 | 最も単価が高いものを代表とする |

## 技術確認チェックリスト（これだけ聞く）

**webアクセスログがインプットに含まれ、かつアウトプットにweb行動由来のカラムがある場合のみ質問する。**
web行動キーワード: 閲覧、カート、お気に入り、PV、Click、サイト

### T1. リレーション項目の確認
- [ ] アウトプットで使うデータ（商品ID等）はどのリレーション項目に格納されているか？
- [ ] そのリレーション項目のデータ形式は？（1レコード1値 / カンマ区切り複数値）
  - **1レコード1値** → 分割不要、そのまま名寄せ・ランキング可
  - **カンマ区切り** → 分割→テンプレート変換が必要。最大件数も確認

### T2. 結合キーの確認
- [ ] webアクセスログと顧客テーブルを結合する際のリレーション項目番号は？（通常リレーション項目_1=顧客ID）
- [ ] webアクセスログのデータと商品テーブルを結合する際のキーは？（リレーション項目_N = 商品ID）

### T3. カート・お気に入りの取得方法（該当する場合のみ）
- [ ] パターンA: カート一覧ページ/お気に入り一覧ページのリレーション項目から取得？
  - → カンマ区切り複数値の可能性高い
  - → **カート商品IDが格納されているリレーション項目は何番か？**（顧客IDとは必ず別番号）
- [ ] パターンB: 商品詳細ページのイベント（イベント発生要素_○○）から取得？
  - → 1レコード1値、絞込み条件に「イベント発生要素_○○ = 特定値」が追加
  - → どのイベント発生要素のどの値で判定するか？
  - → **カート商品IDが格納されているリレーション項目は何番か？**（顧客IDとは必ず別番号）

> **注意**: 顧客IDのリレーション項目番号（T2で確認）とカート商品IDのリレーション項目番号は**必ず異なる**。同一カラムに両方を格納することは物理的に不可能。処理ステップのStep1で両方のリレーション項目を同時に選択する（一方でも漏れるとエラー）。

---

## 質問が不要な場合

以下の場合はT1〜T3の確認も不要。質問せず即座に設計書JSONを出力する：
- webアクセスログがインプットに含まれない（bizデータだけで完結）
- アウトプットにweb行動由来のカラムがない

---

## 質問のルール

- 質問は必ず3つの選択肢を提示する（A/B/C形式）
- **1回の応答で1つの質問に絞る**
- ユーザーの回答を受けて次の質問に進む
- すべて確認できたら設計書JSONを出力する
- **要件に関する質問は絶対にしない**（対象期間、件数、優先順位、除外条件等）
