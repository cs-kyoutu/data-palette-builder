# Webアクセスログ系パターン

## リレーション項目のNULL文字列クレンジングパターン

webアクセスログのリレーション項目に文字列"null"が入っている場合の対処：
1. カラム名変更: リレーション項目_N → リレーション項目_N-算出用
2. 複製: リレーション項目_N-算出用 → リレーション項目_N-書式変換用
3. 書式変換: 書式変換用を半角英小文字に変換（"NULL"→"null"統一）
4. IF文: 書式変換用が"null"に完全一致→空白、それ以外→算出用の値
5. カラム名変更: IF文結果をリレーション項目_Nに変更

リレーション項目_1〜10に対して繰り返し（10回×3操作=30ステップ）。

## IDマッピング統合テンプレート

b→dashの統合テンプレート機能でwebアクセスログ・受注・メール行動ログを紐づけ：

```
『統合テンプレート』
webアクセスログに【{web_log_file}】を選択し、必要なカラムを次の通りに選択する
webアクセスログID: 「{web_log_id}」
ビジターID: 「{visitor_id}」
PV/Click日時: 「{pv_click_datetime}」
ページURL(パラメータ付き): 「{page_url}」
受注ID: 「{relation_order_id}」
顧客ID: 「{relation_customer_id}」
受注に【{order_file}】を選択し、必要なカラムを次の通りに選択する
受注ID: 「{order_id}」
顧客ID: 「{customer_id}」
```

## セグメント作成手順

データパレット構築後にセグメント作成が必要な場合の手順：

```
『セグメント作成』
手順1. セグメント一覧から[新規作成]を押下し、[カスタマイズ][単一セグメント]を選択する
手順2. データファイルに【{data_file}】を選択する
手順3. 必要なカラムに「{key_column}」を選択する
手順4. 絞り込み条件に「{filter_column}」が{filter_condition}と設定する
手順5. セグメント名に""{segment_name}""と入力して保存
```

## アクション別ビジターID振り分け→個別IDマッピングパターン

1つのwebアクセスログから複数アクション（カート/お気に入り/商品詳細等）の集計を同時に行う場合：

### 手順
1. 絞込み: 対象期間+対象ページURLで絞る（OR条件で複数アクション含む）
2. IF文×N回: ページURL条件別にビジターIDを別カラムに振り分け
   - cart含む→カート_ビジターID、bookmark含む→お気に入り_ビジターID、等
3. 集約: 振り分けたカラムをキーに含めて集約
4. 横統合×N回: 各ビジターIDカラムをキーにIDマッピングと結合→顧客ID、顧客ID(1)、顧客ID(2)
5. 最終集約: 年月をキーに各カラムをユニークカウント

### ポイント
- 1つのアクセスログから複数指標を同時算出できる（別々にファイルを作らなくていい）
- 顧客ID(1)、顧客ID(2)は横統合の順番で自動付番される
- 各アクションのビジターID数と顧客ID数を同時に取得可能

## ID紐づけ後の顧客ID確定パターン

IDマッピング統合テンプレートで顧客IDを紐づけた後、リレーション項目の顧客IDとIDマッピングの顧客IDの両方が存在する場合がある。
**IF文で優先判定して1つの顧客IDカラムを作成する**：

```
『IF文』
「リレーション項目_1」が《空文字ではない》の場合、「リレーション項目_1」に変換
いずれの条件分岐にも該当しない場合、「顧客ID【IDマッピング】」に変換
→ 作成したカラムを「顧客ID」に変更する
```

この確定した「顧客ID」で顧客テーブルと横統合する。

## Web接客行動ログの配信日時復元パターン

Web接客のimpressionイベントの日時を、後続のclickイベントにも付与する：
1. IF文: 行動タイプがimpression→行動日時、それ以外→空白 → 「配信日時_パーツ1」
2. 参照(前の行の値): ビジターID+セッションID+施策IDでグループ化、行動日時昇順、配信日時_パーツ1を参照（空文字/Null無視しない）→ 「配信日時_パーツ2」
3. IF文: 配信日時_パーツ1が空文字でない→パーツ1、それ以外→パーツ2 → 「配信日時」

## N回目のアクション日時を取得するパターン

「3回目の閲覧日時」「2回目の購入日」等、特定のN回目のアクションを取得する場合：

1. 絞込み: 対象期間・対象ページ等で絞る
2. 集約: ユーザーID×日時×対象で集約（同一セッション内の重複を残す/排除する制御）
3. ランキング: ユーザーIDでグループ化、日時の小さい順、同率なし → 訪問順位
4. 絞込み: ランキング = N（例: 3）で絞込み
5. カラム名変更: 日時カラムを「N回目閲覧日時」にリネーム
6. 削除: ランキング等の中間カラムを削除

名寄せ（最新/最古の1件）との違い: N番目を正確に指定できる。

## 同時閲覧商品の相関算出パターン（N:N自己結合）

「この商品を見た人はこんな商品も見ています」を作る場合：

### 手順概要
1. **閲覧データ準備**: webアクセスログから閲覧商品IDがあるレコードを抽出、セッションID×商品IDで集約
2. **N:N自己結合**: 同じファイルを自分自身と横統合（セッションIDをキー、《統合処理をエラーにせず実行する》）
   - 左: 「閲覧商品ID」、右: 「同時閲覧商品ID」としてカラム名変更
3. **販売中商品のみに絞込**: 商品マスタと横統合（INNER JOIN）して存在確認
4. **同時閲覧数の集計**: 閲覧商品ID×同時閲覧商品IDで集約、同時閲覧商品IDをカウント
5. **同一商品除外**: N:N自己結合後、左右に同じ商品IDカラムができる（例: 商品IDと商品ID(1)）。IF文で「商品ID」が「商品ID(1)」《次のカラムの値に完全一致》→"1"、それ以外→空白。絞込みで"1"以外を残す
6. **ランキング**: 閲覧商品IDでグループ化、同時閲覧数の大きい順で上位N件

### 最新セッション売上上位商品を起点にする場合
1. 参照で最新セッションIDを取得
2. IF文で最新セッションの閲覧商品のみ抽出
3. 売上データと横統合→売上金額でランキング→1位絞込
4. 上記の相関データと横統合

### 併買商品の相関算出（受注ID×受注ID）
同じ受注内で一緒に購入された商品の組み合わせを作る場合も同じ構造：
1. 受注明細を受注ID×商品IDで集約
2. 同じファイルを自分自身と横統合（受注IDをキー、《統合処理をエラーにせず実行する》）
3. 左: 「購入商品ID」（=商品ID）、右: 「併買商品ID」（=商品ID(1)）
4. 同一商品除外: IF文で「商品ID」が「商品ID(1)」《次のカラムの値に完全一致》→"1"、それ以外→空白 → 絞込みで"1"以外を残す
5. 併買数集計 → ランキング → 上位N件

閲覧版との違い: キーがセッションID→受注ID、閲覧商品ID→購入商品IDに変わるだけで構造は同じ。

### N:N自己結合の注意
- 同じファイル同士の横統合は《統合処理をエラーにせず実行する》を選択
- 多対多結合でレコード数が爆発するため、事前に集約でレコードを減らしておく
