# 運用・管理系パターン

## 差し替え前データ（プレースホルダーテーブル）パターン

将来のデータ連携に備えて、まだ存在しないカラムを先にテーブル定義しておく場合：
- 必要なデータ型のカラムを1つずつ用意したCSVを作成しアップロード
- 横統合で他のデータに結合（LEFT JOINでNULLが入る）
- 実データが来たら差し替え前データを更新するだけ

---

## 既存データの修正パターン

既に構築済みのデータファイルに対してカラム追加・修正を行う場合：
- インプットの素材が「既存」→ 既にb→dash内にあるデータファイル
- 加工は最小限（連結1つだけ等）
- 修正後のデータでセグメントを差し替えるケースが多い

手順:
1. 既存データファイルを加工して新規ファイル作成
2. 新規ファイルでセグメントを新規作成
3. 既存の複合セグメントで旧セグメントを新セグメントに差し替え

---

## 加工元差し替え+タスク編集パターン

既存の加工済みファイルの加工元データや条件を変更する場合：

```
【ファイル名】
　1.加工元を【新しい加工元ファイル】に差し替える
　2.以下加工タスクを上書き保存で編集する。
　2-1
　AS IS
　『絞り込み』
　「カラム」が""旧値""《条件》に絞り込む
　TO BE
　『絞り込み』
　「カラム」が""新値""《条件》に絞り込む
```

AS IS / TO BEで変更前後を明示。変更のない加工タスクは記載不要。

---

## 統合設定の部分変更パターン

既存の統合ファイルを複製して、一部設定だけ変更する場合：

```
【新ファイル名】
　1.【既存ファイル】を複製する。
　2.統合設定を変更する。
　※残すカラム以外は既存と同じ設定
　『横統合』
　（変更する設定のみ記載）
```

---

## IF文+絞込みによる差分抽出パターン

2つのデータソースを横統合した後、一致しないレコードだけを抽出する場合：

1. 横統合: LEFT JOINで2ソースを結合
2. IF文: 「カラムA」が「カラムB」《次のカラムの値に完全一致》→"1"、不一致→「カラムA」の値
3. 絞込み: IF文結果が"1"に完全一致しないレコードを残す

これで「ソースAにあってソースBにないレコード」が抽出できる。
b→dashの絞込みにはカラム同士の比較がないため、IF文で判定フラグを作る必要がある。

用途: 配信不達リストの差分、退会者の除外、重複チェック等

---

## クレンジングタスクの削除パターン

既存の加工済みファイルのタスクを一部削除して再構成する場合：

```
《本加工データファイルを編集し上書き保存》を押下
『クレンジングタスクの削除』
7『集約』、8『四則演算』、9『抽出』を削除
```

タスク番号と操作名を指定して不要なタスクを削除。その後に新しいタスクを追加する。

## 年月マスタ動的生成+欠損月補完パターン

レコードが存在しない月に架空のレコードを補完する場合：

### パーツ①: 年月マスタの動的生成
1. 名寄せ: 顧客IDで1件に絞り込み（代表レコード取得）
2. 追加: 加工処理実行日カラム（日付型）
3. 抽出: 先頭5文字で年を取得（例: "2026/"）
4. テキスト挿入: 文末に"01/01"→ "2026/01/01"
5. 型変換: 日付型に変換
6. 時刻演算: カスタムで過去12ヶ月分（-12〜-1）と未来11ヶ月分（+1〜+11）の日付カラムを作成
   - 「本日年月」-12 → 昨年1月、-11 → 昨年2月、...、-1 → 昨年12月
   - 「本日年月」= 今年1月（基準）
   - 「本日年月」+1 → 今年2月、+2 → 今年3月、...、+11 → 今年12月
7. テンプレート「横持ちを縦持ちに変換」: 24ヶ月カラムを縦持ちに（残すカラム: 顧客ID）
8. 抽出: 縦持ちカラムの先頭7文字で年月を取得

### パーツ②: 元データの年月抽出
1. 抽出: 元データの日付カラムの先頭7文字で年月を取得

### パーツ③: 年月マスタ×元データの横統合
1. 横統合: パーツ①×パーツ②を顧客ID+年月をキーにLEFT JOIN
   → 存在しない月はNULLレコードとして残る

### 最終データ: NULL補完
1. IF文: 年月がNULL（=元データにない月）→ 金額カラムに0を代入、日付カラムに縦持ちカラムの値を代入
2. 型変換: 補完した値を適切な型に変換
3. 削除: 中間カラム削除
4. カラム名変更: 統合で生じた重複カラム名を修正

## 地域分割パターン（絞込みの件数制限対応）

カンマ区切り複数指定の絞込みで値が多すぎる場合、2グループに分割：
1. パーツA: 地域グループ1で絞込み → 保存
2. パーツB: 地域グループ2で絞込み → 保存
3. 最終統合: パーツAとパーツBを横統合 or 縦統合

b→dashの絞込みで「次を含む(カンマ区切りで複数指定)」に入れる値の件数が多い場合に有効。

## SQLジョブによるデータ加工パターン

GUI操作では実現困難な計算をSQLで実現する場合：

### 使うケース
- PARTITION BYによるウィンドウ関数（受注内按分等）
- 複雑な四則演算のチェーン（6カラム以上の加減算）
- CASE WHEN + SUMの組み合わせ

### 手順書の記載方法
```
受注明細データ（ID:XXXX）のSQLをコピーする
以下の対応を行うSQLを作成
①【インプットデータ】からカラムを復活させる
②四則演算を追加（SQL内で実装）
③按分計算を追加（PARTITION BY + ROUND）
```

### 按分計算の典型SQL
同一受注番号内で小計が0より大きいレコードのみに割引額を按分：
```sql
CASE
    WHEN 小計 > 0 THEN
        ROUND((CASE WHEN 小計 > 0 THEN 小計 ELSE 0 END)
             * 原本_割引額
             / SUM(CASE WHEN 小計 > 0 THEN 小計 ELSE 0 END)
               OVER (PARTITION BY 受注番号), 0)
    ELSE 0
END AS 按分額
```

### 注意
- SQLジョブはb→dashのデータパレット内で実行（Snowflake SQL）
- 既存のSQLジョブをコピーして編集するのが一般的
- GUIの加工とSQLジョブは混在可能（SQLで作ったデータを後続のGUI加工でさらに加工）

## 既存データファイルの複製→参照先差替えパターン

既存の統合・加工済みファイルのパーツを新バージョンに差し替える場合：
1. 既存ファイルを複製
2. 統合の参照元（パーツ）を新バージョンに差替え

手順書では「【既存ファイル】を複製後、【旧パーツ】を【新パーツ】に差替え」と記載。
