# 高度な統合・パイプラインパターン

## 差分蓄積パイプライン（前日データとの比較）

前日のデータと当日を比較してステータス変化を検知する場合、b→dashのSFTP連携で「前日スナップショット」を作成する。

### 手順
1. **連携システム作成**: データパレット → データを連携する → 新規追加 → 基幹 → SFTP → 受取
2. **出力先作成**: 出力先 → 新規追加 → 基幹 → SFTP → 送付 → 連携システムの接続情報を入力（ディレクトリパスは「input」）
3. **SSH公開鍵交換**: 出力先で発行された公開鍵を連携システムに貼り付け
4. **データ出力設定**: 対象ファイル → データ出力 → 外部システム出力 → 出力先選択 → 全列 → 繰り返し毎日02:00 → ヘッダー出力
5. **取込設定**: 連携システム → 出力ファイル選択 → 取込 → 定期更新（全削除全更新）

### 用途
- ステータス変化検知（公開/非公開、在庫あり/なし等）
- 前日との差分抽出
- 日次更新リストの作成

## 多段横統合パターン（6テーブル以上の段階的結合）

横統合は2ファイルずつしかできないため、複数テーブルを段階的に結合：
1. A × B → 中間①
2. 中間① × C → 中間②
3. 中間② × D → 中間③
4. ...
各段階で保存名を付けて、次の統合のインプットにする。

## 時系列自己結合パターン（LTV・月次推移等）

同じデータの過去N月分を横に並べる場合：

### 手順
1. **集約**: 年月単位で集計（稼働ユーザー数、売上合計等）
2. **日付カラム作成**: 注文年月+"/01"をテキスト挿入→日付型に型変換
3. **時刻演算**: 日付-1ヶ月、-2ヶ月...で「Nか月前の日付」カラムをN個作成
4. **自己結合**: 自分自身と横統合。左=「Nか月前_日付」、右=「注文年月日」をキーに
5. **N回繰り返し**: 1か月前→2か月前→...→Nか月前まで段階的に横統合
6. **カラム名変更**: 重複カラムを_1、_2...にリネーム
7. **NULL→0置換**: LEFT JOINでNULLになった月を0に
8. **型変換**: テキスト→整数型
9. **チェーン加算**: 直近Nか月の合計を四則演算の連鎖で算出

### 注意
- 自己結合は横統合の特殊ケース（同じファイル同士）
- NULL置換+型変換が大量に発生する（カラム数×月数分）
- 四則演算は2カラムまでなので、N個の合計はN-1回の加算が必要

## ダミー統合用CSVによる多段統合パターン

大量のカラムを持つデータを段階的に統合・加工する場合：

### 構造
1. 加工: 既存ファイルをクレンジング初期化（追加→削除）+ IF文で重複カラム優先判定
2. 統合: 加工結果×ダミー統合用CSVをLEFT JOIN（顧客ID = 統合用カラム）
3. 加工: 続きのIF文
4. 統合: 結果×ダミー統合用CSV
5. ...繰り返し

### ダミー統合用CSVとは
- 「統合用カラム」1カラムだけのCSVファイル
- 全レコードが空（またはNULL）
- LEFT JOINすると元データがそのまま残り、ダミー側は全カラムNULL
- 目的: 統合することで新しいデータファイルが作られ、次の加工の起点にできる

### 重複カラムの優先判定IF文
複数ソースから同名カラムが来る場合（例: 苗字漢字, 苗字漢字(1), 苗字漢字(2)）：
```
『IF文』
「苗字漢字_更新後」が空文字以外の場合、「苗字漢字_更新後」に変換。
「苗字漢字 (2) (1)」が空文字以外の場合、「苗字漢字 (2) (1)」に変換。
「苗字漢字 (2)」が空文字以外の場合、「苗字漢字 (2)」に変換。
それ以外の場合、「苗字漢字 (1)」に変換。
```
最新のソース（更新後 > (2)(1) > (2) > (1)）を優先する。

## 既に縦持ちのデータにマスタ統合→横持ち変換パターン

既存の縦持ちデータ（物件一覧等）にマスタ情報を横統合してから、テンプレートで横持ちに変換するパターン：

1. **横統合**: 縦持ちデータ × マスタを統合（キー: 物件ID等）→ マスタ情報を付与
2. **型変換**: 必要なカラムの型変換（個別に上書き保存）
3. **テンプレート 縦→横変換**: 顧客ID等を集約キー、物件情報カラムを横並び、ソート順を指定
4. **カラム名変更**: (N)付きカラムを①②③等にリネーム

ポイント: マスタ統合を**横展開の前**に行うことで、横統合が1回で済む。横展開後にマスタ統合するとN回必要になり非効率。

## 既存の加工済みデータファイルをインプットにするパターン

b→dash内で既に加工済みのデータファイル（順位付きリスト等）をそのままインプットにする場合：
- インプットのテーブル定義はb→dash内のファイル名とID（例: 「【順位3】空いたら登録者かつ対象物件統合データ(665)」）
- 新規のテーブル定義アップロード不要
- ファイル名に含まれる数字IDはb→dashのデータファイルID

## 参照「後ろの値」によるCV判定パターン

カート投入後に購入（CV）したかを判定する場合：
1. 参照（後ろの行の値）: 顧客IDでグループ化、PV/Click日時の昇順ソート、CVフラグを参照（空文字/Null無視）
2. 絞込み: 参照結果が空文字（=カート投入後にCVしていない）のレコードを残す

これでカート投入したが購入していないユーザーを特定できる。

---

## 参照「前の行の値」による再予約判定パターン

同一顧客が過去にキャンセルしたプランを再予約したかを判定：
1. 参照（前の行の値）: 顧客ID+商品ID+コースIDでグループ化、申込日の昇順ソート、予約キャンセルフラグを参照
2. 前の行がキャンセル（True）→ 現在の行は再予約

## 参照「後ろの行の値」で翌月継続判定パターン

月次のアクティブ率（開封率、購入継続率等）を算出する場合：

### 手順
1. 絞込み: 対象アクション（開封、購入等）に絞る
2. 抽出+テキスト挿入+型変換: 日時から年月を抽出→"/01"付与→日付型に変換
3. 集約: 顧客ID×配信年月でユニークカウント
4. 参照（後ろの行の値）: 顧客IDでグループ化、配信年月の昇順ソート、配信年月を参照 → 「次回アクション月」
5. 時刻演算: 次回アクション月 - 当月 = 経過月数
6. IF文: 経過月数=1 → "1"（=翌月も継続）、それ以外→空白
7. 型変換: 整数型に変換
8. 時刻演算: 当月+1ヶ月 → 集計用の翌月カラム

### 用途
- メール開封の翌月継続率
- 購入の翌月リピート率
- ログインの翌月継続率

## 全顧客共通データの統合パターン（統合用カラム固定値）

顧客個別ではなく全体共通のデータ（人気商品ランキング等）を全顧客に紐づける場合：

1. ランキングデータ側: 追加で固定値"1"の「統合用カラム」を作成
2. 顧客データ側: 同様に追加で固定値"1"の「統合用カラム」を作成
3. 横統合: 統合用カラム同士をキーに → 全顧客に同じランキングデータが紐づく

### 注意
- 顧客側でも集約（統合用カラム+顧客IDをキー）してから統合することで、不要カラムを削減
- 統合後は統合用カラムを削除

---

## 商品分類リストによる分類別処理パターン

商品にカテゴリ分類を付与して分類ごとに別々の処理を行う場合：

1. 商品分類リスト（マスタ）を手動アップロード
2. 閲覧/購入データと商品分類リストを横統合（商品IDをキー）
3. 分類ごとに絞込み（例: 「分類」が"ビューティー"に完全一致）
4. 分類ごとにランキング→横展開→カラム名変更（接頭辞に分類名を付与）

※ヘルス/フード等の他分類も同じ手順で作成し、最後に統合。
