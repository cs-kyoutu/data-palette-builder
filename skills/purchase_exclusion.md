# 購入済み除外ロジック

配信タイミングによって除外方法が変わる。必ず「配信タイミングはいつですか？」を最初に確認する。

---

## 当日中の配信（カート投入後N時間以内等）

bizデータ（受注等）は1日1〜3回の同期のため、リアルタイムでの購入済み判定ができない。

→ **webコンバージョンデータ**で除外する。

### webコンバージョンデータとは
b→dashの仕様で、特定のURL到達やページ内要素のクリックイベントをCV（コンバージョン）として別テーブルに切り出す機能。
**当日内にリアルタイム更新される**ため、日次バッチの受注データを待たずに購入判定が可能。

### カラム構成
| カラム名 | データ型 | 説明 |
|---------|---------|------|
| webコンバージョンID | テキスト | webアクセスログIDとコンバージョンIDを結合したID |
| webアクセスログID | テキスト | webアクセスログデータファイルのID |
| ビジターID | テキスト | webサイトの訪問者ごとに付与されるID |
| コンバージョン時刻 | 日時 | コンバージョン発生時刻（yyyy/MM/dd hh:mm:ss） |
| コンバージョンID | テキスト | コンバージョン設定のID |
| コンバージョン名 | テキスト | コンバージョン名称（例: 購入完了ページ到達） |
| コンバージョンタイプ | テキスト | pageview（PV到達）/ click（クリックイベント） |

### 処理フロー
1. **webコンバージョンデータを絞込み**: コンバージョン名が「購入完了」等のCV名で絞込み + コンバージョン時刻がカート投入時刻以降
2. **カート投入リストと横統合**: ビジターIDをキーにLEFT JOIN
3. **IF文で購入フラグ作成**: コンバージョン時刻がNULLでなければ購入済みフラグ=1
4. **絞込み**: 購入済みフラグ=0のレコードのみ残す

### ユースケース
- カート投入から3時間後にリマインドメール → 日次の受注データ同期を待てないのでwebコンバージョンで判定
- リアルタイムトリガー配信 → 同上

### 注意事項
- webコンバージョンはビジターID単位のため、顧客IDとの紐づけにはIDマッピング（テンプレート ID紐づけ）が必要
- コンバージョン名はテナントごとに設定が異なるため、ユーザーに「購入完了のコンバージョン名」を確認する必要がある

## 翌日以降の配信

bizデータが同期済みなので、**受注データ x 受注明細データ**で正確に判定できる。こちらの方が正確。

### 処理フロー

1. 受注 x 受注明細を横統合して顧客ID + 商品IDのペアを取得（直近N日以内に絞込み）
2. カート投入リストと顧客ID + 商品IDの2キーで横統合（LEFT JOIN）
3. 購入リスト側がNULLでなければ購入済み → 除外

### 注意事項
- 受注データは1日1〜3回の同期のため、配信直前の購入は反映されない場合がある
  - 例: 翌日朝に配信する場合、前日夜〜当日朝の購入が同期されていない可能性がある
  - 運用上の許容範囲として認識しておくこと（処理フロー自体の変更は不要）

## ヒアリング質問例

「購入済みユーザーの除外は必要ですか？配信タイミングはいつですか？」
- A) 当日中（N時間以内）→ webコンバージョンデータで除外
- B) 翌日以降 → 受注データ x 受注明細で除外（商品ID単位で正確に判定）
- C) 除外不要
